Designated Tutor (지정튜터) Feature

PRD & Wireframes — v0.1 draft

Status: Draft v0.1 Date: 2026-05-12 Author: podo@day1company.co.kr
Working repo: personal_prd Touch repos: podo-app, podo-backend, grape

1. Goal

Let students browse, favorite, and book classes with specific PODO tutors they like — parallel to (not replacing) the current subscription-based "any available tutor" booking flow.

The current product matches students to any available tutor on a chosen time slot (ScheduleTimeBlock aggregated by cnt per time). Today, students can only exclude tutors via le_tutor_exclusion — there is no way to actively pick or favorite a tutor, and no public tutor profile.

This PRD introduces the inverse: discovery (profile + search), preference (favorites), commerce (지정튜터 티켓), and a tutor-aware booking flow.

Review data source decision (needs alignment)

Today, lesson-review collects a 1–5 NPS rating tied to classId + tutorId and stores it privately. To populate public reviews:

  • Option A (lean): Aggregate existing NPS scores as public star average; show free-text only when student opts in. Lower content volume early, but no moderation risk.
  • Option B (rich): Add a separate "튜터 리뷰" step in lesson-review funnel, mandatory star + optional text, default-public with opt-out. Needs moderation pipeline (admin in grape).

Recommend Option A for v1 → reuse existing NPS, add opt-in field. Saves a moderation system from being built before we know if students will write reviews.

7. Things the user did NOT mention but matter

  1. Audio bio recording UX. Who records it? Recommend tutors self-record in apps/tutor-web, capped at 30 sec, with admin moderation hold.
  2. Tutor capacity guard. If 200 students all favorite the same star tutor, the slot grid will be ♥0 most of the time. Need a "favorite a backup too" nudge.
  3. Favorite cap. Mirror tutor-exclusion's 30-cap.
  4. Favorite ↔ block conflict resolution. Bidirectional clear on either action.
  5. Lesson cancellation / penalty rules for ticket-booked lessons. Recommend identical rules.
  6. Subscription co-existence. When user has both, default to "지정튜터 티켓" only when a tutor is selected.
  7. Refund of unused tickets when tutor quits. Tickets are generic (constraint C4) → non-issue.
  8. Notification on favorite tutor opens a slot. Powerful retention lever — flag as v1.1.
  9. Public reviews moderation. Option A defers this risk.
  10. Tutor profile privacy of realName. Must never leak to student-facing API; only name.
  11. Schedule API payload size. Aggregated favorite-count for grid, full tutor list lazy-loaded on tap.
  12. Analytics events:
    • tutor_search_viewed, tutor_search_filtered
    • tutor_profile_viewed { tutorId, source }
    • tutor_audio_played { tutorId, duration_played }
    • tutor_favorited / tutor_unfavorited
    • designated_ticket_purchased { sku, amount }
    • booking_favorite_toggle, tutor_picker_opened, designated_booking_completed
  13. Internationalization. All copy through existing i18n pipeline.
  14. Cross-platform parity. apps/native consumes the same Next.js webview.
  15. A/B & rollout. Wire under TBD_2606_DESIGNATED_TUTOR → 5% → 25% → 100%.

10. Definition of done (v1, phase 1 + 2)

  • A user with no past lessons can: search tutors → open a profile → listen to bio → read 5 reviews → favorite → buy a 5-ticket bundle → return to booking → toggle "찜한 튜터만" → tap a slot → pick a favorited tutor → confirm booking → see the booked tutor on /reservation.
  • A user with past lessons sees the new tutor-led card on /reservation, with their own star rating shown when a review exists.
  • Existing booking flow (no toggle, no ticket) is unchanged for users not in the rollout.
  • All new endpoints behind TBD_2606_DESIGNATED_TUTOR flag.
  • Tutor realName does not appear in any student-facing response (test enforced).
