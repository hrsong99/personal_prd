Designated Tutor (지정튜터) Feature

PRD & Wireframes — v0.1 draft

Status: Draft v0.1 Date: 2026-05-12 Author: podo@day1company.co.kr
Working repo: personal_prd Touch repos: podo-app, podo-backend, grape

1. Goal

Let students browse, favorite, and book classes with specific PODO tutors they like — parallel to (not replacing) the current subscription-based "any available tutor" booking flow.

The current product matches students to any available tutor on a chosen time slot (ScheduleTimeBlock aggregated by cnt per time). Today, students can only exclude tutors via le_tutor_exclusion — there is no way to actively pick or favorite a tutor, and no public tutor profile.

This PRD introduces the inverse: discovery (profile + search), preference (favorites), commerce (지정튜터 티켓), and a tutor-aware booking flow.

List structure — favorites pinned on top

  • The list opens with a 찜한 튜터 section showing up to ~5 of the user's favorited tutors (most-recently-favorited first). A "전체 보기 →" link in the section header deeplinks to /my-podo/tutor-favorites for full management.
  • Below that, 전체 튜터 contains everyone else (excluding tutors the user has already favorited, to avoid duplicates). Sort + filter chips apply only to this section.
  • When the user has zero favorites, the 찜한 튜터 section is hidden entirely — the list just shows 전체 튜터 from the top with no extra chrome.
  • Same structure when the page is reached from the booking flow (see §4.4.2): the slot context is dropped, so the user is exploring the full list, but their favorites still pin top.

Card design rationale

  • Tutor name is the title — bumped to 17px bold. Other text drops a tier in weight so the eye lands on the name first. Country flag sits inline as a smaller secondary mark next to the name.
  • Heart anchored to the card's top-right corner (absolute-positioned), not the top of the content column. Position stays fixed regardless of how much content sits in the column — predictable touch target.
  • 72px square portrait (rounded corners) anchors the card with a real face — borrowed from Langdy. Tutor.profileLargeImage already exists, so this is a render-side change only.
  • Audio overlaid on the portrait as a 🔊 icon in the bottom-left, instead of a separate play button competing for space on the right. Tapping plays in place; the icon flips to the primary green while playing. Hidden when there's no audio bio yet.
  • NEW badge — orange pill in front of the tutor name. Driven by Tutor.hireDate < 30d ago. Cheap merchandising lever for fresh tutors to accumulate reviews faster.
  • "신규" rating treatment — new tutors show ★ 신규 instead of ★ 0.0. Avoids the credibility-killing "zero stars" misread when the truth is "no data yet." Switches to a numeric rating once review count ≥ 5.
  • Language level inline — "한국어 가능" / "영어만" sits on the rating row, not as a separate hashtag chip. Treated as an attribute of the tutor, not a tag they earned.
  • "함께한 레슨 N회" on its own metadata row in green when N≥1; muted "함께한 적 없음" when 0. This is PODO's unique signal vs. competitors and earns its own line.
  • Heart vs. bookmark — kept the heart pictogram, preserves the favorite ↔ exclusion mental model PODO already has (tutor-exclusion is the inverse).
  • Resting (휴식 중) tutors are hidden from the list entirely. A tutor with classPause=true is filtered out of /tutors server-side. Reasoning: a tutor you can't actually book this week wastes a slot on the discovery surface, and dimming them in-place doesn't deprioritize them as effectively as removing them. They reappear automatically when 휴식 ends. (Edge: if a user has already favorited a resting tutor, the favorite stays in /my-podo/tutor-favorites with a "휴식 중" pill — they don't get silently unfavorited.)

Empty / edge states

  • No results → "검색어와 일치하는 튜터가 없어요" + suggest clearing filters.
  • Tutor canUse=false, canTakeClass=false, or classPause=true → hidden from list entirely.
  • Favorited tutor goes on rest (classPause=true after they were favorited) → stays in /my-podo/tutor-favorites with a "휴식 중" pill; just removed from search and from booking-screen ♥N counts.

Review data source decision (needs alignment)

Today, lesson-review collects a 1–5 NPS rating tied to classId + tutorId and stores it privately. To populate public reviews:

  • Option A (lean): Aggregate existing NPS scores as public star average; show free-text only when student opts in. Lower content volume early, but no moderation risk.
  • Option B (rich): Add a separate "튜터 리뷰" step in lesson-review funnel, mandatory star + optional text, default-public with opt-out. Needs moderation pipeline (admin in grape).

Recommend Option A for v1 → reuse existing NPS, add opt-in field. Saves a moderation system from being built before we know if students will write reviews.

Favorite-available chip design

  • Slots where at least one favorited tutor is available get a pink border + light pink tint on the chip — no number, no inline icon.
  • Count is intentionally dropped: the user only needs to know "yes/no," not "how many." Three vs. one favorited tutor at 12:00 doesn't change the user's decision; either way they're free to pick.
  • Chip dimensions stay identical across the entire grid — the only difference is color.
  • Top-right of the time slots gets a small two-item legend — pink heart "찜한 튜터" + gray dot "마감" — mirroring the existing 예약 마감 legend pattern.
  • Disabled (마감) slots are never marked favorite — that color combination would be misleading.
  • Users with no favorited tutors don't see pink chips at all → zero visual noise added.

Slot tap behavior

  • Tap on any slot — pink or plain — opens the same existing BookingConfirmDialog.
  • The dialog gets one new row, "튜터 선택하기", only when the slot is pink (i.e., favorites available) and the user owns at least one 지정튜터 ticket (see §4.4.2).

§4.4.2 — Tap a slot → confirm dialog (with optional tutor picker entry)

The booking confirm dialog gets one new row: "튜터 선택하기". It appears only when (a) at least one of the user's favorited tutors is available at the chosen slot and (b) the user owns at least one 지정튜터 ticket. Otherwise, the dialog is identical to today.

Where the picker row leads

Tapping "튜터 선택하기" closes the dialog and navigates to /tutors (§4.1) — the standard tutor list, with favorites pinned at the top. No separate picker sheet, no slot-scoped filtering. The user picks a tutor, opens that tutor's profile, taps "이 튜터로 예약하기", and chooses their preferred slot from the slide-up booking modal (§4.2).

Why route through the tutor list instead of an inline picker

  • One discovery surface. The same screen serves "I'm browsing" and "I'm picking right now" — fewer patterns to maintain, easier for users to predict.
  • The user gets the full profile (bio, audio, reviews, history) before committing a ticket, which a 5-row picker sheet can't deliver.
  • Slot context is informational, not load-bearing. The "(찜한 튜터 3명 가능)" copy tells the user that this exact slot is achievable. If they pick a tutor whose schedule doesn't actually include this slot, the profile's schedule grid + slide-up modal handle it gracefully — they pick a different time.
  • Empty/edge cases:
    • User has favorites but 0 tickets → row reads "지정튜터 티켓 필요 — 티켓 구매하기 →" and navigates to /tutors/purchase-ticket instead.
    • User has tickets but 0 favorites → row is hidden (nothing to select from).
    • Both 0 → row is hidden.

Trade-off worth flagging: routing away from the booking dialog means the user loses the slot they had selected. Backed off intentionally — if they come back without picking a tutor, they re-enter the booking flow naturally. If we ever want to "remember the slot," it can be a deeplink (/tutors?intent=designated&slot=…) later. v1 stays simple.

Per-slot tutor count source

ScheduleTimeBlock already stores tutor_id. New endpoints:

GET /api/v3/podo/schedule/{classId}/tutors?date=…&time=…
→ [{ tutorId, name, profileImage, rating, lessonsWithMe, isFavorite }]

GET /api/v3/podo/schedule/{classId}/favorite-counts?date=…
→ { "09:00": 3, "09:30": 1, … }

4.6 — 지정튜터 티켓 purchase /tutors/purchase-ticket

Pricing model

Constraints we're solving for:

  • Tutor cost — English ≈ ₩2,000/lesson, Japanese ≈ ₩3,500/lesson (PODO's tutor payout).
  • Floor: minimum 10% premium on tutor cost, even on the deepest bulk tier.
  • Premium signal — designated booking is a premium upgrade, so 1-ticket price should anchor noticeably above the subscription per-lesson cost.
  • Bulk incentive — meaningful discount at 5/10/20 회권 to push commitment, without ever breaking the 10% floor.

Margin logic

  • 1회권= 100% markup over tutor cost. Anchors the perceived value of a single designated lesson at 2× and lets every bulk tier feel like a discount from a strong reference price.
  • Bulk tiers step down the markup: +100% → +70% → +45% → +20%. The shopper sees a steeper "1회권 대비" discount (−15% / −28% / −40%) but PODO still nets a healthy margin everywhere.
  • 20회권 sits at +20% — twice the required 10% floor, leaving room for promo/coupon discounts (e.g. 신규 가입 −10%, 친구 초대 −5%) without breaking the rule.
  • Cross-language consistency: Japanese sells for exactly 1.75× English at every tier, the same ratio as tutor cost (₩3,500 / ₩2,000), so margin percentage is identical across languages. This makes copy + ops easier (one discount schedule).
  • SKU floor guardrail — backend rejects ticket SKUs whose price/total_count falls below tutor_cost × 1.10, enforced at SKU creation in grape.

"가장 인기" anchor

Recommend marking 10회 as the featured tier (vs. 5회 in earlier draft). 10회 is the sweet spot: −28% discount feels substantial, but the wallet hit (₩29k EN / ₩50k JP) stays under the typical 1-month subscription, so it's not a competing bet. 5회 ends up acting as the "try it" entry, 20회 as the power-user option.

Ticket data model

le_designated_tutor_ticket
─ id
─ student_id
─ sku                   (e.g. EN_10, JP_5)
─ language              ('EN' | 'JP')   ← NEW vs earlier draft
─ total_count
─ remaining_count
─ purchased_at
─ expires_at            (purchased_at + 90d)
─ refunded_at           NULLABLE
─ source_order_id

The language column means: an English ticket can only book English-language tutors, and vice versa. This is enforced at registerWithDesignatedTutor() by joining Tutor.language against the ticket's language. Prevents accidentally burning a ₩5,000 Japanese ticket on a ₩2,000-cost English lesson.

  • C4 — Ticket specificity: recommend generic (tutor chosen at booking time). Language-locked, but not tutor-locked.
  • C5 (new) — Promo stacking: can the 신규 가입 −10% coupon stack with the 20회권 −40% bundle? Margin floor +20% says yes, but final check belongs to Finance.

7. Things the user did NOT mention but matter

  1. Audio bio recording UX. Who records it? Recommend tutors self-record in apps/tutor-web, capped at 30 sec, with admin moderation hold.
  2. Tutor capacity guard. If 200 students all favorite the same star tutor, the slot grid will be ♥0 most of the time. Need a "favorite a backup too" nudge.
  3. Favorite cap. Mirror tutor-exclusion's 30-cap.
  4. Favorite ↔ block conflict resolution. Bidirectional clear on either action.
  5. Lesson cancellation / penalty rules for ticket-booked lessons. Recommend identical rules.
  6. Subscription co-existence. When user has both, default to "지정튜터 티켓" only when a tutor is selected.
  7. Refund of unused tickets when tutor quits. Tickets are generic (constraint C4) → non-issue.
  8. Notification on favorite tutor opens a slot. Powerful retention lever — flag as v1.1.
  9. Public reviews moderation. Option A defers this risk.
  10. Tutor profile privacy of realName. Must never leak to student-facing API; only name.
  11. Schedule API payload size. Aggregated favorite-count for grid, full tutor list lazy-loaded on tap.
  12. Analytics events:
    • tutor_search_viewed, tutor_search_filtered

  13. Internationalization. All copy through existing i18n pipeline.
