# PRD: 그레이프 어드민 SMS 발송 기능

## 개요

현재 내부 팀에서는 그레이프 어드민 내에서 특정 학생에게 일회성 SMS를 보낼 방법이 전혀 없습니다 — CS 에스컬레이션, 결제 리마인드, 또는 카카오 알림톡 템플릿이 적합하지 않은 모든 상황에서요. 본 PRD는 학생 상세 페이지에 단일 진입점 형태의 "문자 보내기" 기능을 추가하고, 그 흐름을 감사(audit) 가능하고 유지보수 가능하게 만드는 데 필요한 어드민 페이지들을 함께 추가합니다.

이 기능은 새로운 통합을 만드는 게 아니라 **기존 NHN Cloud SMS 발송 코드(`inc/podo_sms_sender.php`)를 래핑**합니다. 그레이프는 이미 NHN과 SMS, LMS, 알림톡으로 통신하고 있으며, 본 PRD는 그 위에 UI 기반 발송 경로와 감사 로그를 얹는 것입니다.

이것은 **무언가의 대체가 아니라 신규 어드민 기능**입니다. 기존 알림톡 플로우(`admin/marketing/manual_alimtalk.php`)와 기존 프로그램 방식의 SMS 호출(영업 프로세스, 결제 플로우 등)은 변경되지 않습니다.

---

## 목표

1. 학생 상세 페이지(`admin/podo_students.php?USER_ID=X&mode=U`)에서 그레이프 어드민 유저라면 누구나 "문자 보내기" 팝업을 열어 해당 학생에게 2~3회 클릭만으로 SMS를 보낼 수 있습니다.
2. 변수 치환(`{{student_name}}`, `{{phone}}`, `{{user_id}}`, `{{수강권_name}}`)이 지원되는 템플릿으로 일관된 메시지를 빠르게 보낼 수 있으며, 본문은 발송 전까지 자유롭게 수정할 수 있습니다.
3. 90바이트 SMS 한도를 초과하는 메시지는 LMS로 자동 업그레이드되며, UI 뱃지로 요금 티어가 표시되어 어드민이 인지할 수 있습니다.
4. 모든 발송 기록(수신자, 본문, 발신 번호, 발송한 어드민, 발송 상태)이 남으며, "발송 로그" 페이지에서 필터(날짜, user_id, user_name, 발송 어드민)로 조회할 수 있습니다.
5. SMS 템플릿과 발신 번호는 별도 CRUD 페이지에서 어드민이 직접 관리할 수 있어 코드 배포 없이 셀프 서비스가 가능합니다.
6. 본 기능은 그레이프(PHP) 내부에서 완전히 완결됩니다 — podo-backend(Kotlin)는 관여하지 않습니다.

## V1 비목표 (Non-Goals)

- **추가 진입점 없음.** "문자 보내기" 버튼은 오직 학생 상세 페이지에만 노출됩니다. 대량 발송 없음, 목록 페이지 발송 없음, 예약 발송 없음.
- **MMS / 이미지 첨부 없음.** 텍스트 전용.
- **역할 기반 권한 게이팅 없음.** 모든 그레이프 어드민이 발송 및 로그 조회 가능. (기존 그레이프 접근 제어로 충분.)
- **팀별 / 역할별 기본 발신 번호 없음.** V1은 전사 공통 기본 번호 1개.
- **신규 podo-backend 서비스 / API 없음.** 그레이프가 기존 래퍼를 통해 NHN을 직접 호출.
- **보관 정책 / 자동 삭제 없음.** 발송 로그 레코드는 영구 보관.
- **딜리버리 콜백용 웹훅 엔드포인트 없음.** 메커니즘은 개발 단계로 위임(V1은 폴링 우세, §10 참고).
- **템플릿 변수로 이메일 없음.** 변수는 위 4종 픽스드 숏리스트로 한정.
- **변수 네임스페이스 확장 없음.** 위 4종 변수만 치환되며, 그 외는 그대로 리터럴로 남음.

## 용어집

| 용어 | 의미 |
|---|---|
| **그레이프 어드민** | `grape.re-speak.com/admin/`의 내부 PHP 어드민 도구. 절차적 PHP 7+, mysqli, Bootstrap 3 + jQuery. |
| **NHN Toast SMS** | NHN Cloud의 SMS/LMS 서비스. 엔드포인트 형식: `https://api-sms.cloud.toast.com/sms/v2.3/appKeys/<appKey>/sender/{sms,lms}`. 인증 정보는 `.env`의 `CONF_TOAST_APP_KEY` / `CONF_TOAST_SECRET_KEY`에 저장. |
| **SMS** | 단문, ≤ 90바이트(한글 ~45자 / 영문 ~90자). 저렴한 요금. |
| **LMS** | 장문, > 90바이트 ~ 약 2,000바이트(한글 ~1,000자). 다른 NHN 엔드포인트, SMS 대비 약 3배 요금. |
| **발신 번호** | NHN에 사전 등록된, SMS의 "보낸 사람" 필드에 사용 가능한 번호. 그레이프는 이 번호들을 신규 DB 테이블에 저장. |
| **템플릿 변수** | 템플릿 본문에 들어가는 `{{...}}` 플레이스홀더. 어드민이 템플릿을 선택하는 순간 학생 데이터로 치환됨. |
| **발송 어드민** | 발송을 트리거한 그레이프 유저 계정 (`$_SESSION['user_admin_id']`, `$_SESSION['user_admin_email']`). |

## 유저 스토리

| 누가 | 무엇을 원하는가 | 이유 |
|---|---|---|
| 학생 페이지를 보는 어드민 | 한 번의 클릭으로 "문자 보내기" 버튼을 누르고 싶다 | 페이지를 떠나지 않고 학생에게 문자를 보낼 수 있다 |
| 발송 팝업의 어드민 | 학생의 이름, user_id, 휴대폰이 자동 채워지고 읽기 전용으로 보이길 원한다 | 잘못된 사람에게 보낼 위험이 없다 |
| 발송 팝업의 어드민 | 드롭다운에서 발신 번호를 고르고 싶다(합리적인 기본값과 함께) | 어떤 번호가 등록되어 있는지 외울 필요가 없다 |
| 발송 팝업의 어드민 | 드롭다운에서 템플릿을 고르면 본문이 학생 데이터로 치환된 채 자동으로 채워지길 원한다 | 일관되면서도 개인화된 메시지를 몇 초 만에 보낼 수 있다 |
| 발송 팝업의 어드민 | 템플릿 적용 후에도 본문을 수정할 수 있길 원한다 | 특정 학생에 맞게 문구를 다듬을 수 있다 |
| 긴 메시지를 보내는 어드민 | SMS 한도를 넘으면 글자수 카운터와 "LMS" 뱃지를 보고 싶다 | 더 비싼 티어로 발송됨을 알고, 줄일지 그대로 보낼지 결정할 수 있다 |
| 보내기 버튼을 누르는 어드민 | SMS가 실제로 나가기 전 확인 다이얼로그를 보고 싶다 | 실수 클릭으로부터 보호받는다 |
| 발송 실패 상황의 어드민 | 팝업 안에 인라인 에러와, 현재 본문을 그대로 재사용하는 "재시도" 버튼이 뜨길 원한다 | 다시 입력하지 않고 즉시 복구할 수 있다 |
| 과거 발송을 감사하려는 어드민 | "발송 로그" 페이지를 열고 날짜 / user_id / user_name / 발송 어드민으로 필터링하고 싶다 | "이 사람한테 이미 보냈나?" 또는 "어제 Alice가 뭘 보냈지?" 같은 질문에 답할 수 있다 |
| 템플릿을 관리하는 어드민 | 전용 페이지에서 템플릿을 추가/수정/소프트 삭제하고 싶다 | 팀이 코드 배포 없이 큐레이션된 템플릿 라이브러리를 유지할 수 있다 |
| 발신 번호를 관리하는 어드민 | 전용 페이지에서 발신 번호를 추가/수정/기본 지정/활성 토글하고 싶다 | NHN 등록 번호가 바뀔 때 팀이 직접 처리할 수 있다 |

## 기능 요구사항

### 1. 진입점: 문자 보내기 버튼

- **"문자 보내기"** 버튼을 학생 상세 페이지 상단 버튼 줄에 추가 (`admin/podo_students.php`, 약 1170~1173번째 줄, "포도계정 생성" / "QA 계정 일괄 생성" 옆).
- 클릭 시 **Bootstrap 모달 팝업**이 같은 페이지 내에서 열림(다른 인라인 액션 패턴, 예: `notification_manual_send.php`와 동일). `window.open` 없음, 페이지 이동 없음.
- 모달은 페이지의 `$_REQUEST['USER_ID']`를 키로 사용.

### 2. 발송 팝업 구성

모달은 위에서 아래로 다음 요소를 렌더링합니다.

1. **수신자 블록 (읽기 전용)**
   - 이름 (student_name)
   - User ID
   - 휴대폰 (정규화된 번호)
2. **발신 번호 드롭다운** — `SMS_SENDER_NUMBER` 테이블에서 `is_active = 1`인 행 목록; `is_default = 1`인 행이 미리 선택됨.
3. **템플릿 드롭다운** — `SMS_TEMPLATE` 테이블에서 `is_active = 1`인 행 목록. 템플릿 선택 시:
   - 템플릿 본문을 메시지 textarea에 로드
   - 지원되는 각 변수를 학생 데이터로 치환
   - V1 변수 셋: `{{student_name}}`, `{{phone}}`, `{{user_id}}`, `{{수강권_name}}`
   - 해당 학생의 변수 소스 필드가 null/빈 값이라면, 본문에 리터럴 `{{...}}` 플레이스홀더가 그대로 남아 어드민이 인지하고 수정할 수 있게 함
   - 인식되지 않는 변수(고정 숏리스트 외)는 리터럴로 남음 — 에러 없음
4. **메시지 textarea** — 편집 가능. 기본 상태는 비어 있음(템플릿 미선택); 템플릿을 고르면 덮어씀.
5. **글자수 카운터 + 티어 뱃지** — 현재 바이트 수와 "SMS" / "LMS" 뱃지를 함께 표시:
   - 0~90 바이트 → **SMS** 뱃지
   - 91 바이트 이상 → **LMS** 뱃지 (호버 시 작은 "LMS — 약 3배 요금" 힌트)
   - 2,000 바이트에 도달하면 발송 버튼 비활성화
6. **발송 버튼** — 확인 다이얼로그(§3) 트리거.
7. **취소 버튼** — 모달을 닫고 작성 중인 내용을 버림.

### 3. 확인 다이얼로그

- 발송 버튼 클릭 후, 수신자의 이름과 휴대폰이 보이는 두 번째 모달이 **"정말 발송하시겠습니까?"**를 묻습니다.
- 버튼: **발송** (기본), **취소** (보조).
- **발송** 클릭 시에만 실제 SMS가 나갑니다.

### 4. 발송 메커니즘

- PHP 발송 핸들러는 NHN 호출을 새로 구현하지 않고 **`inc/podo_sms_sender.php`를 래핑**합니다.
- 래퍼는 본문 길이에 따라 NHN 엔드포인트를 선택:
  - ≤ 90 바이트 → SMS 엔드포인트 (`/sender/sms`)
  - > 90 바이트 → LMS 엔드포인트 (`/sender/lms`)
- 만약 `podo_sms_sender.php`가 현재 SMS만 노출한다면, 래퍼에서 LMS 엔드포인트 지원을 추가해야 함. (구현 시 기존 코드 리뷰로 범위 확정.)
- 인증 정보는 `.env`(`CONF_TOAST_APP_KEY`, `CONF_TOAST_SECRET_KEY`) 재사용. 신규 인증 정보 없음.
- NHN HTTP 성공 시, `SMS_SENT_LOG` 테이블에 `pending` 상태로 행을 기록(아직 배송 미확정).
- NHN HTTP 실패 시(비-2xx, 네트워크 에러, 타임아웃), `api_failed` 상태로 행을 기록하고 NHN 에러 코드/메시지를 저장, **그리고** 팝업에 인라인 에러를 표시하며 "재시도" 버튼을 노출. 재시도는 현재 폼 값을 그대로 다시 제출; 새 드래프트 상태 불필요.

### 5. 발송 로그 페이지

- 신규 페이지: `admin/sms/sent_log.php` (정확한 경로는 개발자가 결정 — §8 참고).
- `inc/admin_sidebar.php`의 신규 최상위 **SMS** 섹션에서 링크.
- 기본 뷰: 가장 최근 50건 발송, 페이지네이션 적용.
- **필터** (조합 가능):
  - 발송 날짜 범위 (from / to)
  - User ID (정확 일치)
  - User Name (LIKE)
  - 발송 어드민 (한 번이라도 발송한 어드민 드롭다운)
  - 발송 상태 (sent / delivered / failed / pending / api_failed)
- **컬럼**:
  - 발송 일시 (created_at)
  - 보낸 사람 (발송 어드민 email/name)
  - 받는 사람 (학생 이름 + user_id + 휴대폰)
  - 발신 번호
  - 본문 (말줄임표, 클릭 시 펼침)
  - 유형 (SMS / LMS)
  - 발송 상태 (delivered / failed / pending / api_failed)
  - 템플릿 (사용 시 템플릿 이름, 아니면 "직접 작성")
- 각 행은 읽기 전용; V1에서는 로그 페이지에서 수정/재발송 없음.

### 6. SMS 템플릿 관리 페이지

- 신규 페이지: `admin/sms/templates.php` (경로 미정).
- SMS 사이드바 섹션에서 링크.
- 목록 뷰 표시 항목: 템플릿 이름, 본문 미리보기, 생성자, 마지막 수정자, 마지막 수정 일시, 활성 여부.
- 필터: 활성만 보기(토글), 이름 검색.
- CRUD 액션: **추가** (모달), **수정** (모달), **비활성화** (소프트 삭제: `is_active = 0`).
- 소프트 삭제된 템플릿은 발송 팝업의 드롭다운에서 사라지지만, 로그에서는 FK를 통해 이름이 그대로 조회됨(§9.2 참고).
- 변수 picker UI: 템플릿 본문을 편집할 때, 지원되는 4개 변수 토큰을 삽입할 수 있는 "변수 추가" 헬퍼 노출.

### 7. 발신 번호 관리 페이지

- 신규 페이지: `admin/sms/sender_numbers.php` (경로 미정).
- SMS 사이드바 섹션에서 링크.
- 목록 뷰: 발신 번호, 설명/라벨, 기본 여부, 활성 여부.
- CRUD: **추가**, **수정**, **활성/비활성 토글**, **기본 지정** (라디오; 하나를 기본으로 설정하면 다른 행의 기본 플래그가 원자적으로 해제됨).
- `is_active = 1`인 번호만 발송 팝업 드롭다운에 노출.
- 임의 시점에 `is_default = 1`인 행은 정확히 하나(앱 코드에서 강제, §9.1 참고).

### 8. 사이드바 네비 배치

- `inc/admin_sidebar.php`에 신규 최상위 **SMS** 섹션, 3개 서브 링크:
  1. 발송 로그 (sent log)
  2. 템플릿 관리 (templates)
  3. 발신 번호 관리 (sender numbers)
- 신규 섹션은 기존 최상위 섹션(마케팅, 영업 등)과 형제 관계; 정확한 순서는 개발자가 결정.

## 기존 그레이프 코드와의 통합

### 10. 재사용, 재구축 아님

- 신규 발송 핸들러는 (기존) `inc/podo_sms_sender.php`를 호출합니다. 기존 함수/클래스를 확장하거나 래핑하여 다음을 지원:
  - 명시적 발신 번호 인자 수용 (현재는 단일 기본값에 의존할 수 있음)
  - 본문 바이트 길이에 따라 NHN의 SMS 또는 LMS 엔드포인트로 라우팅
  - 호출자가 `SMS_SENT_LOG`를 채울 수 있을 만큼의 정보 반환 (NHN request_id, 상태 코드, 에러 정보)
- 인증/세션: 발송 핸들러는 `inc/check_admin.php`가 설정하는 `$_SESSION['user_admin_id']` / `$_SESSION['user_admin_email']`을 신뢰; 별도 인증 플로우 없음.
- DB: `inc/db_conn.php`의 기존 mysqli 커넥션 사용.
- `podo_sms_sender.php`에 현재 인라인되어 있는 하드코딩된 `appKey`(코드 스캔에서 발견된 보안 노트)는 본 PRD의 **범위 외**지만, 후속 작업을 위해 flag — 인증 정보는 `.env`에만 존재해야 함.

### 11. 스키마 배포

- 스키마 변경은 그레이프 컨벤션에 따라 `admin/sql/` 디렉터리의 SQL 파일로 배포:
  - `create_sms_sender_number_table.sql`
  - `create_sms_template_table.sql`
  - `create_sms_sent_log_table.sql`
- 시드 데이터: `SMS_SENDER_NUMBER`에 최소 1행, `is_default = 1` 및 `is_active = 1`. (별도 `seed_sms_sender_number.sql`로 번들하거나 배포 후 DevOps가 적용.)

### 12. 발송 상태 — 개발자에게 위임

PM은 발송 상태 메커니즘을 개발자에게 위임했습니다. 두 가지 유효한 접근 방식:

- **폴링 (V1 권장)**: cron 작업이 N분마다 실행되어, 최근 24시간 내 `delivery_status = 'pending'`인 `SMS_SENT_LOG` 행을 모아 NHN 상태 조회 API를 호출하고 행을 업데이트. 인프라가 더 단순; 퍼블릭 콜백 URL 불필요.
- **웹훅**: NHN에 콜백 URL 등록(그레이프 자체가 `grape.re-speak.com`으로 퍼블릭하게 접근 가능), 콜백을 수신하는 얇은 PHP 엔드포인트를 노출하고 행을 업데이트. 실시간에 가깝지만 엔드포인트 하드닝 + 서명 검증 필요.

**메커니즘과 무관한 스펙 계약**: `delivery_status` 컬럼은 합리적인 시간 내(목표: 1시간 미만, 하드 천장: 24시간)에 모든 행에 대해 `pending` → `delivered` 또는 `failed`로 이동해야 함. 24시간이 지나도 `pending`에 멈춰 있는 행은 로그에서 시각적으로 surfac됨.

## 인수 조건 (Acceptance Criteria)

그레이프 어드민이 다음을 할 수 있어야 합니다:

1. **팝업 열기** — `podo_students.php?USER_ID=X&mode=U`에서 "문자 보내기" 클릭 시, 학생의 이름/user_id/휴대폰이 자동 채워지고 읽기 전용으로 표시됨.
2. **발신 번호 선택** — `SMS_SENDER_NUMBER`의 `is_default = 1` 행이 기본 선택된 드롭다운에서 발신 번호 선택.
3. **템플릿 선택** — `is_active = 1` 템플릿 드롭다운에서 선택 시, 본문이 학생 레코드의 `{{student_name}}`, `{{phone}}`, `{{user_id}}`, `{{수강권_name}}` 값으로 정확히 치환되어 자동 채워짐.
4. **빈 변수는 리터럴 유지** — 변수 소스 필드가 null/빈 값인 경우, 본문에 리터럴 `{{...}}` 플레이스홀더가 그대로 남음.
5. **본문 편집** — 템플릿 적용 후에도 본문 편집 가능.
6. **LMS 뱃지** — 본문이 90바이트를 초과하면 LMS 뱃지가 노출되고, 그 이하에서는 SMS 뱃지가 노출됨.
7. **발송 → 확인 → 실제 전송** — 보내기 클릭 시 "정말 발송하시겠습니까?" 확인 모달이 뜨고, 발송 클릭 시 실제로 SMS가 나감.
8. **로그 기록** — `SMS_SENT_LOG`에 새 행: `sent_by_admin_id` = 본인 admin id, `body` = NHN이 실제로 받은 렌더링된 텍스트, `message_type` = 선택된 엔드포인트와 일치하는 SMS/LMS, `delivery_status` = 초기 `pending`.
9. **API 실패 처리** — NHN API 실패 시 팝업에 인라인 에러 + 재시도 버튼이 표시되며, `SMS_SENT_LOG`에 `api_failed` 행 기록.
10. **로그 필터링** — 발송 로그 페이지에서 각 필터(날짜 범위, user_id, user_name, 발송 어드민, 발송 상태)를 개별/조합 적용 시 결과가 올바르게 제한됨.
11. **템플릿 CRUD + 소프트 삭제 동작** — 템플릿 추가/수정/소프트 삭제 가능, 소프트 삭제된 템플릿은 발송 팝업에서 사라지지만 과거 로그 행에서는 이름이 계속 보임.
12. **발신 번호 CRUD + 원자적 기본 변경** — 발신 번호 추가, 기본 지정(이전 기본은 원자적으로 해제됨), 활성/비활성 토글 가능.
13. **영구 보관** — 발송 로그는 자동 삭제 없이 영구 보관.
14. **상태 자동 전이** — `pending` 행이 폴링 윈도우(또는 웹훅) 내에 `delivered`/`failed`로 바뀌며, 어드민 액션 없이 로그 행에 반영됨.

## 열린 가정 (유저와 검증됨)

- 그레이프는 NHN Cloud (Toast)를 SMS 프로바이더로 사용하며, 인증 정보는 `.env`(`CONF_TOAST_APP_KEY`, `CONF_TOAST_SECRET_KEY`)에 존재. ✓ 코드베이스 스캔으로 확인.
- NHN은 동일 appKey/secret으로 SMS와 LMS를 모두 지원. (NHN Cloud SMS 제품 표준.)
- 학생 상세 페이지에는 변수 치환을 위한 `student_name`, `phone`, `user_id`, `수강권_name` 데이터 소스가 이미 접근 가능. (페이지가 이미 해당 필드를 렌더링 중.)
- LMS는 NHN에서 메시지당 SMS 대비 약 3배 요금.
- 그레이프 어드민 로그인 신뢰 모델로 발송에 충분 — 발송 시 추가 확인 요소(예: 비밀번호 재입력)는 불필요.
