## 1. 문제 정의 (Problem)

프로모션·이벤트 기간에 맞춰 구매 보너스 정책(`PURCHASE_BONUS_TICKET_COUNT`, `PURCHASE_BONUS_EXTENSION_DAYS`)을 시작/종료 시각에 정확히 변경해야 하지만, 현재는 어드민의 즉시 수기 변경(`sys_code_detail_update.php`)만 가능하다. 이 때문에:

- 프로모션 시작·종료 시각에 운영자가 **대기조 형태로 깨어 있어야** 한다 (특히 자정/심야 케이스).
- 사람이 직접 누르는 만큼 **누락·지연·오타** 위험이 상시 존재한다.
- "프로모션 종료 후 원복"이 매번 운영 부담으로 돌아온다.

## 2. 목표 (Goals)

1. 어드민이 두 코드의 detail 값을 **미래 KST 시각**에 자동으로 적용되도록 사전 등록할 수 있다.
2. 등록된 예약은 **운영자 개입 없이 ±1분 내** 적용된다.
3. 적용 결과(성공/실패)가 **Slack 운영 채널에 즉시 통보**된다.
4. 등록·취소·실행 이력이 모두 DB에 남아 **사후 추적**이 가능하다.

### Non-Goals (v1 범위 외 — 명시적으로 안 한다)

- 일반 시스템 코드 전체에 대한 범용 스케줄러 (대상 코드 화이트리스트로 한정)
- 자동 원복(window) — "X시각부터 Y시각까지" 단일 폼
- 예약 수정(편집) — 취소 후 재생성으로 갈음
- 승인/결재 워크플로우
- podo-backend 미리보기 diff
- Slack 외 다른 채널 알림 (이메일, SMS 등)

## 3. 사용자 (Users)

| 페르소나 | 사용 목적 | 빈도 추정 |
|---|---|---|
| 운영팀 어드민 | 프로모션 시작/종료 시각에 보너스 정책 자동 변경 | 주 1–3회 (프로모션 캘린더 기준) |
| 개발팀 (간접) | Slack 알림으로 실패 인지 → 수동 복구 | 실패 시점에만 |

**권한 모델:** 별도 role을 신설하지 않는다. **시스템 코드 편집 권한이 있는 일반 admin**이면 누구나 등록·취소 가능. (→ Open Question Q-6)

## 4. 사용자 시나리오 (User Stories)

- **US-1 (등록)**: 운영자로서, 4월 30일 자정에 6개월 결제 티켓 보너스를 8개 → 10개로 바꾸고 싶다. 미리 폼에서 등록해두고 잠을 자고 싶다.
- **US-2 (묶음 등록)**: 운영자로서, 같은 시각에 티켓 카운트와 연장일수를 동시에 변경하고 싶다. 한 번의 폼 제출로 두 코드를 함께 등록한다.
- **US-3 (취소)**: 운영자로서, 어제 등록한 예약이 잘못된 값임을 발견했다. PENDING 상태인 동안 취소하고 새로 등록한다.
- **US-4 (실패 인지)**: 운영자로서, 새벽에 적용된 예약이 어떤 이유로든 실패했다면 Slack에서 즉시 알고 수동 복구 작업을 시작한다.
- **US-5 (이력 조회)**: 운영자로서, 지난주에 적용된 변경 내역(누가, 언제, 무엇을 바꿨는지)을 한 화면에서 확인한다.

## 5. 기능 요구사항 (Functional Requirements)

### FR-1. 예약 등록
- 어드민 메뉴 "시스템 코드 → 구매 보너스 예약 → 새 예약"에서 진입.
- **입력**: 실행시각(KST, 분 단위), 변경 항목 1개 이상 (코드 / 월수 / 새 값).
- **검증**:
  - 실행시각 > 현재시각(KST) — 과거/현재 시각 등록 불가
  - 동일 분에 PENDING 예약 없어야 함 — 충돌 시 인라인 에러 (`이미 같은 시각에 예약된 항목이 있습니다 (#SCHEDULE_ID 보기)`)
  - 모든 새 값이 비어있지 않아야 함
  - (코드, 월수) 조합이 한 예약 내에서 중복 금지
- 검증 통과 시 schedule + items를 한 트랜잭션에 insert.

### FR-2. 예약 목록
- 컬럼: 예약시각(KST) / 변경 항목 수 / 상태 / 작성자 / 실제 실행시각 / 실패 사유
- 필터: 전체 / PENDING / EXECUTING / DONE / FAILED / CANCELED
- 행 클릭 시 항목 상세(코드 / 월수 / old → new) 표시
- PENDING 행에만 "취소" 버튼 노출

### FR-3. 예약 취소
- PENDING 상태에서만 가능
- 취소 사유 입력은 선택
- 상태 → CANCELED, CANCEL_USER / CANCEL_DATETIME 기록
- cron이 이미 실행 중인 EXECUTING 상태는 취소 불가

### FR-4. 자동 실행
- 매분 1회 cron 트리거 (`* * * * *`)
- PENDING & EXECUTE_AT ≤ NOW() 인 schedule을 EXECUTE_AT 오름차순 순차 처리
- schedule 단위 트랜잭션: 한 schedule의 일부 항목만 적용되는 일은 없다
- 실행 시 `TB_SYS_CODE_DETAIL.CODE_DETAIL_VALUE`의 현재 값을 OLD_VALUE에 스냅샷 후 NEW_VALUE로 update
- 적용 후 `inc/podo_cache_util.php`와 동일한 캐시 무효화 호출

### FR-5. 결과 통보
- 성공/실패 시 Slack `le_bot_respeak_operation` 채널에 메시지 전송 (메시지 형식은 워크오더 §Slack 알림 참조)
- 실패 메시지에는 SCHEDULE_ID·예약시각·예외 사유·@channel 멘션 포함

### FR-6. 화이트리스트 강제
- `PURCHASE_BONUS_TICKET_COUNT`, `PURCHASE_BONUS_EXTENSION_DAYS` 외의 코드 이름은 **form 레벨에서도, process 핸들러에서도** 거부
- DB 레벨 제약은 두지 않음 (이름 변경 유연성 위해 핸들러 측 화이트리스트로 제어)

## 6. 비기능 요구사항 (Non-Functional)

| 영역 | 요구사항 |
|---|---|
| 적시성 | 예약시각 도래 후 **60초 이내** 적용 완료 (cron 1분 주기 + 처리 시간) |
| 신뢰성 | 한 schedule의 부분 적용 금지 (트랜잭션 보장). 예외 발생 시 ROLLBACK + FAILED 마킹 + Slack 알림 |
| 동시성 | 같은 분에 PENDING 두 개 금지 (충돌 차단). cron 다중 인스턴스로 인한 중복 실행 방지 (FOR UPDATE 사용) |
| 캐시 일관성 | 적용 즉시 podo-backend가 새 값을 반영해야 함 (캐시 무효화 검증 필수) |
| 감사성 | 작성자 / 실행시각 / OLD↔NEW / 실패 사유 모두 영구 보존 (논리 삭제 금지) |
| 보안 | 어드민 인증된 세션만 접근. PHP form CSRF 보호는 기존 어드민 패턴 준수 |

## 7. UX 요구사항

워크오더 §UX 상세 참조. 핵심:

- 진입점: 기존 시스템 코드 페이지 상단 탭/링크
- 폼은 **반복 행** 패턴으로 항목 추가/삭제 (드롭다운: 코드 → 월수 동적 로딩, 현재값 readonly 표시)
- 목록은 status 필터 + 행 클릭 상세
- 모든 시각 표기는 **KST 명시** (운영자 혼선 방지)

## 8. 데이터 모델

워크오더 §데이터 모델 참조. 신규 테이블 2개:

- `TB_PURCHASE_BONUS_SCHEDULE` (예약 마스터)
- `TB_PURCHASE_BONUS_SCHEDULE_ITEM` (변경 항목)

## 9. 성공 지표 (Success Metrics)

| 지표 | 측정 방식 | 목표 |
|---|---|---|
| 예약 적용 적시성 | `EXECUTED_AT - EXECUTE_AT` 분포 | p95 ≤ 60초 |
| 적용 성공률 | DONE / (DONE + FAILED) | ≥ 99% (월간) |
| 운영 부담 감소 | 보너스 코드 수기 변경 건수 (출시 전후 비교) | 70% ↓ |
| 사용 정착도 | 예약 등록 건수 / 월 | ≥ 4건 (출시 후 1개월) |

## 10. 리스크 & 의존성

| 리스크 | 완화 |
|---|---|
| cron 인프라 장애로 매분 실행 지연 | Slack 실패 알림 + 운영자 수동 적용 폴백 (기존 `sys_code_detail_update.php` 그대로 살아있음) |
| 캐시 무효화 실패로 backend가 옛 값 사용 | QA 체크리스트 §"적용 후 podo-backend가 새 값을 즉시 반영" 항목 강제. → Q-2 참조 |
| 누군가 EXECUTING 중에 DB를 직접 건드림 | FOR UPDATE 락 + Slack 실패 알림으로 감지. v1에서는 운영 정책으로 방지 |
| 화이트리스트 우회 | 폼 + 핸들러 양쪽에서 검증 |

## 11. 미해결 질문 (Open Questions)

> Socratic 인터뷰가 정상 동작했다면 여기서 뽑혔을 항목들. 구현 착수 전 확정 필요.

- **Q-1 (타임존 일관성)** — DB가 KST 가정인지, UTC 저장 후 변환인지? `EXECUTE_AT DATETIME`에 KST 시각을 그대로 넣고 cron의 `NOW()`도 KST라고 가정해도 되나? MySQL `time_zone` 세션 설정 확인 필요.
- **Q-2 (캐시 무효화 실패 처리)** — DB 트랜잭션은 COMMIT 됐는데 `/api/v1/cache/podo/clear/all` 호출이 실패하면? 옵션: (a) FAILED로 마킹하고 알림, (b) DONE으로 마킹하되 별도 캐시 재시도 큐, (c) 그냥 로그만. 현재 워크오더는 (c)에 가까움.
- **Q-3 (지연된 schedule의 적용 정책)** — cron이 1시간 멈췄다가 재개되어 1시간 전 EXECUTE_AT인 schedule이 잡히면? "그래도 적용"인지 "stale로 skip하고 FAILED 마킹"인지. 운영 관점에서 후자가 안전할 수 있음 (이미 프로모션 종료된 변경을 뒤늦게 적용하면 문제).
- **Q-4 (Slack 알림 실패 처리)** — Slack API 장애로 메시지 전송이 실패하면? schedule은 이미 DONE인데 알림만 누락. 재시도 정책?
- **Q-5 (감사 로그 일관성)** — 일반 어드민에서 코드를 바꾸면 어떤 감사/이력 테이블에 흔적이 남는지(예: `TB_LOG_*`)? cron 적용도 동일한 위치에 동일한 형식으로 기록되어야 운영자 조회가 일관됨.
- **Q-6 (권한 식별자)** — "시스템 코드 편집 가능한 admin"의 구체적인 권한 키는? 기존 `sys_code_list.php`의 접근 가드를 그대로 이식하면 되는지 확인.
- **Q-7 (cron 동시 실행 방지)** — `FOR UPDATE`만으로 충분한가? cron 두 인스턴스가 동시에 시작하면 한쪽은 락 대기 후 빈 결과를 받는다. SKIP LOCKED 도입 또는 단일 인스턴스 보장(파일 락) 검토.
- **Q-8 (프로모션 시작/종료 페어링)** — 보통 프로모션은 "시작 + 종료" 두 시각의 한 쌍. v1은 각각 별도 schedule로 등록해야 하는데, 종료 schedule을 깜빡할 위험. UX에서 "쌍으로 등록" 같은 도우미를 둘지, 아니면 이 위험을 운영 가이드로만 처리할지.
- **Q-9 (OLD_VALUE NULL 케이스)** — 실행 시점에 해당 `(CODE_NAME, CODE_DETAIL_NAME)` row가 이미 삭제됐으면? upsert로 갈지, FAILED로 갈지.
- **Q-10 (수동 변경과의 충돌)** — 예약 등록 후, 운영자가 같은 코드를 즉시 변경 페이지에서 손으로 또 바꿔버린 경우 어떻게 해야 하나? cron은 OLD_VALUE를 새로운 현재값으로 스냅샷할 텐데, 작성자 의도와 달라질 수 있음. 폼 제출 시 "예약된 변경 있음" 경고 표시?

## 12. 출시 계획 (Rollout)

1. **개발/스테이징** — 워크오더의 QA 체크리스트 8개 항목 통과
2. **프로덕션 배포** — 새 테이블 마이그레이션 + cron 등록 (서버 crontab)
3. **검증 운영** — 첫 1주는 운영자가 수기 변경과 병행하여 결과 비교 (1건 수동 / 1건 예약)
4. **정착** — 1개월 후 사용 빈도·성공률 측정, Q-3·Q-7 등 미해결 항목을 v1.1 백로그로 이관
